
## Hand-off: UI Migration & Preact Polish (Phase 5.5)

### What was accomplished in this session:
1. **Ported Query Builder:** Successfully converted the legacy vanilla JS `query_builder.js` into a declarative Preact component (`QueryBuilderUI`) embedded directly within the content script's `DashboardTab` (`extension/src/content/ui.js`). Eliminated direct DOM mutation targeting by lifting state up to the parent component.
2. **Refactored Logic Engines & Storage Bindings:** Repaired a disconnect where the new `RulesTab` and `MacrosTab` (in the Options page) were using separate storage keys from the underlying `RuleEngine` and `MacroEngine`. The engines and the UI components now synchronously read/write to `STATE.config.smart_rules` and `STATE.config.batch_macros` and persist via the central `STATE.saveConfig()` wrapper.
3. **Fixed Rules Deletion Bug:** Updated the `RulesTab` delete function to filter rules based on their composite key (`type` and `source`) because rules generated by the `RuleEngine` do not have a unique `id` field.
4. **Restored Context-Dependent UI Tabs:** Reverted an erroneous extraction of the `TemplatesTab` (The Architect) and `GraphTab` (Semantic Graph). Because these features require direct access to the DOM overlay and real-time active tab collections, they must live inside the injected content script (`extension/src/content/ui.js`), not the isolated Options page context. They have been restored and rewired into the main Dashboard.

### Notes to Self / Learnings:
- **Preact Migration Caveat:** When migrating legacy vanilla JS UI code that relies on `document.getElementById()` to read values, remember to convert those input elements to controlled components (e.g., `<input value=${this.state.val} onInput=${e => this.setState({val: e.target.value})} />`).
- **Options Page vs. Content Script:** The separation of concerns is vital. The Options page cannot scrape the active tab or render overlays onto Raindrop.io. Any UI tab that executes an action requiring DOM access must remain in `ui.js`.
